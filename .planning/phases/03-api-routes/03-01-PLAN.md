  <name>Task 1: JWT auth middleware + public routes</name>
  <files>
    worker/src/middleware/auth.ts,
    worker/src/routes/public.ts
  </files>
  <action>
    **worker/src/middleware/auth.ts:**
    - Use the `jose` library (not jsonwebtoken — jose works with Edge runtime)
    - `generateToken(payload: { staffUserId: string; role: string }, secret: string)`: 
      - Create JWT with HS256, 8-hour expiry
      - Payload includes sub, role, iat, exp
      - Returns compact JWT string
    - `verifyToken(token: string, secret: string)`: 
      - Verify and decode using jose
      - Return payload or throw AppError(401)
    - `createAuthMiddleware(secret: string)`:
      - Returns a request handler that checks Authorization: Bearer <token>
      - Extracts and verifies JWT
      - If invalid/missing, returns 401 JSON
      - If valid, attaches staffUser to request context
      - Role-based access: accept optional `allowedRoles: string[]` parameter
      - Throws unauthorized or forbidden appropriately

    **worker/src/routes/public.ts:**
    - `registerPublicRoutes(router: Router, services: { appointmentService, patientService, availabilityService })`:
      - `POST /api/public/appointments/request`:
        - Accept body: { name, phone, email?, serviceType, preferredDate, preferredTimeRange, notes?, source }
        - Validate: name required, phone or email required, serviceType required, preferredDate required, preferredTimeRange required
        - Source auto-set to "web_form" if not provided
        - Step 1: Find or create patient by phone/email
        - Step 2: Check availability for requested date/time range
        - Step 3: If available, send OK response with slot options
        - Step 4: Create appointment with status "requested"
        - Return: { success: true, appointment, patient, availableSlots? }
        - Errors: 400 for validation, 409 for conflict
      - `POST /api/public/appointments/confirm`:
        - Accept body: { appointmentId } or { patientId, startTime, endTime }
        - Validate appointment exists
        - Check still available (not taken by someone else)
        - Update status to "confirmed"
        - Queue notification: confirmation via preferred channel (async — fire and forget)
        - Return: { success: true, appointment }
      - `POST /api/public/appointments/cancel`:
        - Accept body: { appointmentId }
        - Cancel the appointment
        - Notify patient
        - Return: { success: true }
      - `GET /api/public/availability?date=YYYY-MM-DD&serviceType=skin`:
        - Return list of available time slots
        - Return: { date, serviceType, slots: [{ start, end }] }
      - Add OPTIONS handlers for CORS on all routes

    Helper: Extract JSON body from request with try/catch. Use itty-router's `withParams` or manual parsing.

  <action>
    **worker/src/routes/staff.ts:**
    - `registerStaffRoutes(router: Router, services: { all services }, jwtSecret: string)`:
      - All staff routes use auth middleware (with optional role filter)
      
      - `POST /api/staff/auth/login`:
        - Accept: { email, password }
        - Find staff user by email
        - Compare password using bcryptjs (compareSync)
        - If valid, generate JWT via auth middleware
        - Return: { token, staffUser: { id, name, role, email } }
        - Error: 401 for invalid credentials

      - `GET /api/staff/appointments`:
        - Query params: date (default today), status (optional filter)
        - Call appointmentService.getByDate(date, status)
        - JOIN with patients to include patient name/phone
        - Return: { appointments: [...] }

      - `POST /api/staff/appointments`:
        - Accept body with: patientId or { name, phone, email }, serviceType, startTime, endTime, notes
        - If patientId provided, use existing patient
        - If patient info provided, create patient first
        - Set source="staff_manual"
        - Call appointmentService.createManual(...)
        - Return: { appointment }

      - `POST /api/staff/appointments/:id/check-in`:
        - Extract appointment ID from route params
        - Call appointmentService.checkIn(id, staffUserId)
        - Return: { appointment }
        - Note: 400 if status != "confirmed"

      - `POST /api/staff/appointments/:id/complete`:
        - Call appointmentService.complete(id, staffUserId)
        - Return: { appointment }
        - Note: 400 if status != "checked_in"

      - `POST /api/staff/appointments/:id/cancel`:
        - Call appointmentService.cancel(id)
        - Return: { appointment }

      - `POST /api/staff/appointments/:id/no-show`:
        - Set status to "no_show"
        - Return: { appointment }

      - `GET /api/staff/patients/:id`:
        - Fetch patient by ID
        - Fetch their appointment history
        - Fetch their feedback
        - Fetch their follow-ups
        - Return: { patient, appointments, feedback, followUps }

      - `GET /api/staff/patients`:
        - Search patients by name or phone (query param: q)
        - Return: { patients: [...] }

      - `GET /api/staff/follow-ups`:
        - Query params: status (default "pending"), type (overdue|upcoming|all)
        - Call appropriate followUpService methods
        - Return: { followUps: [...] } with patient name joined

      - `POST /api/staff/follow-ups`:
        - Accept: { patientId, reason, dueDate }
        - Validate: patientId exists, reason non-empty, dueDate valid
        - Create follow-up
        - Return: { followUp }

      - `POST /api/staff/follow-ups/:id/complete`:
        - Accept: { notes? }
        - Call followUpService.complete(id, notes)
        - Return: { followUp }

    **worker/src/routes/webhooks.ts:**
    - `registerWebhookRoutes(router: Router, services: { all services }, notificationService)`:
      
      - `POST /api/webhooks/whatsapp`:
        - Twilio webhook format: parse Body, From (wa- formatted), MessageSid, etc.
        - Meta webhook format: parse entry[0].changes[0].value.messages[0]
        - Detect provider based on request body structure
        - Extract: from (phone number), text (message body), messageId
        - Route to conversation handler:
          - If message starts with or matches "book"/"appointment"/"schedule": start booking flow
          - If message contains "cancel": find and cancel upcoming appointment
          - If message is a number 1-5: treat as feedback rating
          - If patient has a pending feedback request: store as feedback text
          - Else: default reply with menu options
        - Return 200 OK (Twilio expects 200 within 15s)
        - For Meta webhook verification: handle GET request with hub.challenge
      
      - `POST /api/webhooks/email-feedback`:
        - Accept: { appointmentId, rating?, comment? }
        - Store feedback
        - Return: { success: true }
      
      - `POST /api/webhooks/whatsapp/status`:
        - Handle delivery status callbacks (optional, ack for now)
        - Return 200 OK
